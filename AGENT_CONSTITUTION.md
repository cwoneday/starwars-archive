# 스타워즈 캐넌 아카이브 — 에이전트 설계 헌법

> 프로젝트 가칭. 연대기 중심 · 디즈니 캐넌 전체 · 매니아 깊이를 목표로 하는
> 스타워즈 대장정 기록 웹페이지의 에이전트 시스템 설계 지침.

---

## 0. 이 문서는 무엇인가

이 문서는 **이 프로젝트 전용 에이전트 설계 헌법**이다. 스타워즈 데이터를 직접
채우는 매뉴얼이 아니라, 에이전트들이 따라야 할 결정의 나침반이다. 작업 중 길을
잃으면 여기로 돌아와 정렬한다.

**기존 자산과의 관계 — 2층 구조.**

- **위층 (이 헌법):** 이 프로젝트 고유의 결정. 연대기 마스터 축, era 표준 enum,
  공유 스키마, 확정도 추적 원칙, 서브에이전트 분할과 전문성 규칙.
- **아래층 (agent-design-framework V2.0):** 이 헌법을 제약 조건으로 받아,
  timeline-curator 같은 개별 에이전트를 PROFILE/RULES/OUTPUT_SCHEMA로 찍어내는
  범용 도구.

헌법이 "무엇을 지켜야 하는가"를 정하고, V2.0가 "그것을 지키는 에이전트를 어떻게
만드는가"를 담당한다. V2.0는 버리지 않는다. 그 위에 한 층을 얹을 뿐이다.
(blog-writing-system과 travel-platform CONSTITUTION이 이미 쓴 것과 같은 판단.)

## 1. 시스템 개요

**마스터 축은 연대기다.** 인물도 행성도 조직도 결국 "어느 사건에 등장했나"로
시간축에 걸린다. 따라서 사건 데이터(events.json)가 시스템의 심장이며, 다른 축은
여기에 매달려 확장된다.

**구조는 오케스트레이터 + 서브에이전트다.** 단일 거대 에이전트로 모든 카테고리를
처리하면 출력이 무너진다(V2.1 검증에서 확인된 "긴 산출물 한 번에 박기" 결함).
종목분석 시스템의 오케스트레이터 + 4축 패턴을 계승한다.

| 구성요소 | 역할 |
| --- | --- |
| 오케스트레이터 | 입력이 사건·인물·행성·관계 중 무엇인지 판단해 적절한 서브에이전트로 라우팅 |
| timeline-curator | **첫 구현 대상.** 사건을 연대·인과 관점에서 스키마로 변환 |
| 인물 프로파일러 | 인물의 동기·관계·변화를 프로파일로 구조화 |
| 관계 추출자 | 자유 텍스트에서 노드-엣지(가족·사제·동맹·적대) 추출 |
| 에피소드 요약자 | 작품 단위 줄거리를 자기 언어로 재구성 (저작권 주의 지점) |

## 2. 헌법 조항

### 제1조 — 연대기가 마스터 축이다
모든 데이터는 시간축에 걸린다. year는 BBY를 음수·ABY를 양수로 하는 단일 숫자
(`year_value`)로 정렬한다. era는 디즈니 8개 시대를 표준 enum으로 고정한다:
제다이의 여명 / 하이 리퍼블릭 / 제다이의 몰락 / 제국의 시대 / 반란의 시대 /
뉴 리퍼블릭 / 퍼스트 오더의 부상 / 뉴 제다이 오더.

### 제2조 — 하나의 공유 스키마, 모두가 따른다
events.json을 비롯한 데이터 스키마는 전 서브에이전트의 공통 계약이다. 어떤
에이전트도 자기 편의로 필드를 바꾸거나 추가하지 않는다. 이것이 일관성을 지키는
바닥이다.

### 제3조 — 그 위에서 각 서브에이전트는 고유 전문 렌즈를 갖는다
같은 사건을 봐도 보는 눈이 달라야 한다. timeline-curator는 "언제·어떤 인과로",
인물 프로파일러는 "동기와 관계가 어떻게 변했나", 관계 추출자는 "누가 누구와 어떤
엣지로"를 본다. 각 에이전트의 PROFILE은 **자기만의 대표 질문 · 자기만의 깊이 ·
자기만의 산출 강점**을 명시해야 한다. 범용 톤으로 수렴하는 것을 금지한다.

### 제4조 — 전문성의 경계를 침범하지 않는다
제3조의 짝. timeline-curator는 인물 심리를 서술하지 않고 사건 사실과 연대에
집중한다. 각 에이전트는 자기 책임 필드만 채우고, 남의 영역은 id 참조로만 가리킨다.
경계 침범은 곧 데이터 충돌이다.

### 제5조 — 확정과 추론을 데이터가 밝힌다
`year_certainty`(exact / approximate / disputed)와
`canon_status`(explicit / referenced / inferred)로, 원작이 확정한 것과 에이전트가
추론한 것을 데이터 스스로 구분한다. 침묵을 확정으로 해석하지 않는다.
(V2.1에서 잡은 "본인 답 vs 추측" 함정의 시스템 버전.)

### 제6조 — 진짜 기록은 자기 언어다
줄거리는 외부 위키에서 복제하지 않는다. 본인이 본 작품을 자기 언어로 재구성한다.
`source` / `source_type` 출처 명시는 필수.
(travel-platform 헌법 5조의 계승.)

### 제6조-A — 화법: 나무위키 톤으로 단정하게
모든 작품(`summary`·`significance`) 서술은 **나무위키식 화법**을 표준으로 한다.
이는 표현 베끼기가 아니라 **화법·정보 밀도의 표준화**다(제6조 위에서 작동).

- **좌표를 먼저 못박는다.** 시리즈 내 위치·작가/감독·발매일을 첫머리에 사실로 박는다.
  ("~를 그린다" 같은 모호한 분위기 서술 금지.)
- **단정적·정보 밀도 높게.** 한 문장에 사실을 빽빽이. 줄거리는 사건의 뼈대를 단정형으로.
- **평가·수용사를 솔직하게.** 호평/혹평이 갈렸으면 갈렸다고, 베스트셀러면 베스트셀러라고 명시.
- **팬덤·제작 디테일을 곁들인다.** 코드명·약칭·별명·제작 비화 같은 "아는 사람만 아는" 맥락.
- 단, 원문 문장·카피·대사는 **절대 복제하지 않는다**(제6조 불변). 화법만 가져온다.

> 기준 템플릿: `light-of-the-jedi`의 summary/significance.

### 제6조-B — 서술은 데이터와 링크로 엮인다
작품·사건 서술에 우리 데이터에 이미 존재하는 **사건·인물·장소·세력·탈것·함선**의
이름이 등장하면, 해당 상세 페이지로 **하이퍼링크를 건다**. 서술이 외딴 텍스트로
끝나지 않고 그물(제1·4조)에 합류하게 하는 장치다.

- 링크는 **표시 페이지(렌더 시점)에서** 데이터의 한국어명을 매칭해 자동 생성한다.
  원본 텍스트(JSON)에는 마크업을 박지 않는다 — 데이터는 순수하게 유지(제2조).
- 매칭은 우리 데이터에 **실재하는 id가 있을 때만** 건다. 없는 이름은 평문으로 둔다(제5조).

### 제7조 — 안전한 점진
8개 시대를 한 번에 채우지 않는다. 시대 하나로 스키마와 에이전트를 검증한 뒤
복제한다. 큰 변경 전 박제, 단계마다 commit + 검증.
(travel-platform 헌법 4조의 계승.)

## 3. 공유 데이터 스키마 (설계 결정 수준)

events.json이 첫 마스터 스키마다. 깊은 필드 정의는 별도 스키마 문서로 두되,
헌법 수준에서는 다음 필드 그룹과 핵심 규칙만 고정한다.

- **식별:** `id`(영문 slug), `title`, `title_en` — 원어 키는 교차참조·외부 대조용 필수
- **연대:** `era`, `year_value`(부호 규칙), `year_label`, `year_certainty`
- **분류:** `category`(battle/politics/duel 등), `tier`(1 주요 / 2 보조 / 3 세부 — 저장은 무한, 표시는 등급별)
- **관계:** `characters`, `locations`, `factions` — 모두 **id 배열 참조** (이름 직접 입력 금지)
- **출처:** `source`, `source_type`(film/series/novel/comic/game), `canon_status`
- **서술:** `one_liner`(미리보기·재사용용 한 줄), `summary`(표면 줄거리), `significance`(심층·의미)
- **인과:** `preceded_by`, `followed_by`(tier 1끼리 먼저), `tags`

> 결정 기록: `spoiler_level`은 두지 않는다 — 매니아 대상, 전면 공개 가정.
> `summary`는 통째로 유지하되 `one_liner` 한 칸만 추가한다.

## 4. 서브에이전트 전문성 매트릭스

제3조·제4조를 실제로 강제하는 표. 각 에이전트 설계 시 이 칸들을 채운다.

| 에이전트 | 전문 렌즈 | 책임 필드 | 침범 금지 영역 |
| --- | --- | --- | --- |
| timeline-curator | 시간·인과 | era, year_*, category, tier, preceded/followed_by | 인물 심리, 관계 해석 |
| 인물 프로파일러 | 동기·성장 | (인물 스키마) 동기, 변화, 소속 이력 | 사건 연대 확정, 전투 전개 |
| 관계 추출자 | 노드-엣지 | (관계 스키마) from/to/type | 줄거리 서술, 의미 부여 |
| 에피소드 요약자 | 작품 단위 서사 | summary, one_liner, source_* | 연대 계산, 관계도 구성 |

각 PROFILE에는 "이 에이전트만 던지는 대표 질문 3개"를 의무 포함시킨다. 전문성이
말로만 끝나지 않고 산출물에 박히게 하는 장치다.

---

## 5. 구현 순서

1. **반란의 시대로 검증.** 사건 수가 적당하고 본인이 가장 잘 아는 오리지널 3부작이라
   "자기 언어 정리"가 자연스럽다. 이 시대로 events.json 스키마가 무너지지 않는지 확인.
2. **timeline-curator를 V2.0로 생성.** 이 헌법(특히 1·2·5·6조)을 제약으로 넣어 PROFILE/RULES/OUTPUT_SCHEMA 도출.
3. **스키마가 안정되면 다른 시대로 복제.**
4. **인물 프로파일러·관계 추출자 순차 추가.** 매번 제3·4조로 전문 렌즈와 경계를 점검.

> 모든 것을 서두를 일이 아니다.
